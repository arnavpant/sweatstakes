# Quick Task 260414-7l8: Migrate Vercel Crons to GitHub Actions Summary

Relocated hourly cron scheduling from `vercel.json` to a free GitHub Actions workflow so the project can deploy on Vercel Hobby tier (which caps crons at 2/day and daily-only).

## What Changed

- **Deleted** `vercel.json`. It contained only the `crons` array — no other Vercel config was preserved (none to preserve).
- **Created** `.github/workflows/cron.yml`. Runs on `schedule: '0 * * * *'` plus `workflow_dispatch`, with two parallel jobs (`settle`, `reminders`), each curling the already-deployed endpoint with `-fsS --retry 3` and the `Authorization: Bearer $CRON_SECRET` header.
- **Zero application code changed.** `src/app/api/cron/settle/route.ts` and `src/app/api/cron/reminders/route.ts` are byte-for-byte identical to before — the new workflow targets their existing Bearer-token gate.

Single atomic commit: `d8a9154 chore(cron): migrate Vercel crons to GitHub Actions for Hobby tier compat` (+58 / -12).

## Operator Action Required BEFORE the Next Push

The workflow will run on its schedule as soon as it lands on the default branch. If either secret is missing it will fail silently (the jobs will run but the curls will 401 or hit an empty host). Set both in GitHub:

**Location:** Settings → Secrets and variables → Actions → New repository secret

1. **`CRON_SECRET`** — MUST exactly match the `CRON_SECRET` env var already set in Vercel. The Vercel-side value is what the route handlers read (`process.env.CRON_SECRET`); the GitHub-side value is what the workflow sends in the `Authorization: Bearer …` header. The two must agree.
2. **`PROD_BASE_URL`** — Full origin of the deployed site with **no trailing slash**. Example: `https://sweatstakes.vercel.app`.

## Vercel `CRON_SECRET` Env Var Is Still Required

Do **not** remove `CRON_SECRET` from Vercel. It's read server-side by both route handlers in `src/app/api/cron/*`. The new GitHub secret of the same name is a separate value living in a separate system — it just needs to match the Vercel value. Rotating it means updating both places.

## Best-Effort Scheduling Trade-off

GitHub Actions scheduled runs are best-effort and can be delayed up to ~15 minutes under high platform load. Accepted because:

- **Settlement cron** does a timezone-aware check for each user's Monday-settlement hour. A delay within that hour still lands in the same window.
- **Reminders cron** buckets per-user reminder windows hourly; a 15-minute delay shifts the notification by the same amount — well within the "worth reminding" tolerance.

This is the documented cost of the free tier.

## No Route Code Changed

`src/app/api/cron/settle/route.ts` and `src/app/api/cron/reminders/route.ts` were deliberately untouched. Their existing `Authorization: Bearer $CRON_SECRET` check is what the new workflow targets. This kept the migration to a pure infra relocation and is why the full 282-test suite still passes without modification.
