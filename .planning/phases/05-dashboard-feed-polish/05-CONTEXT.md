# Phase 5 Context: Dashboard, Feed & Polish

## Phase Goal

All five Stitch-designed screens are fully functional and visually match the Royale design system. This phase completes Dashboard (add stakes + comparative progress + photo gallery), builds Feed from scratch (currently a placeholder), and finishes Settings (add notifications + profile editing).

## Decisions

### Dashboard

**DASH-01 — Active stakes display = "Leader callout" (viewer-POV)**
Show a dynamic motivational line from the current user's perspective:
- If viewer is leading: `"You lead with 12 pts — 3 ahead of Max"`
- If viewer is not leading: `"You're 3 behind Max — closest: Sara at 9 pts"`
- If no settled weeks yet (all balances zero): `"No stakes settled yet — keep checking in"`

Computes from existing `point_transactions` balance (Phase 4). No new schema.

**DASH-02 — Comparative progress = Horizontal card scroll**
Each member of the challenge gets a card showing avatar + name + their own 7-day dot row for the current week. User's own card is pinned first. Horizontal scroll (snap on each card) to see the rest. Reuses `DayDots` component from Phase 3 — one instance per member card.

**DASH-03 — Photo gallery = 2×2 or 3×2 grid preview**
Show the N (6) most recent check-in thumbnails from the current challenge as a tight grid on the Dashboard. Tap a thumbnail navigates to the Feed tab (not an inline lightbox — Feed is the gallery experience).

### Feed

**FEED-01/02 — Layout = BeReal-style vertical card feed**
Full-width cards, one per check-in, scrolled vertically. Each card: avatar + display name + relative timestamp ("2h ago") at top, then the composite check-in photo (workout photo with selfie thumbnail overlay — already generated by the Phase 3 camera flow). No actions below the photo in v1 (no likes/comments; Hype/Nudge deferred to v2).

**Update model = Pull-to-refresh + revalidate on navigation**
No Supabase Realtime subscription in Feed itself. `router.refresh()` runs whenever the user navigates back to the Feed tab, and a pull-down gesture triggers a manual refetch. Keeps Feed implementation lean.

**Pagination = Load all for current challenge**
Query `check_ins` ordered by `created_at DESC` with no `LIMIT`. For friend-group scale (~5 posts/week × 3 months = ~60 rows) this is fine. Revisit if volume grows.

### Settings

**SETT-01 — Weekly goal** — Already implemented in Phase 3 (`goal-stepper.tsx`). No new work.

**SETT-02 — Notifications = Real Web Push via iOS PWA (primary) + Android (free add-on)**

Scope this as its own plan — substantial new infrastructure. Key work items:

- Service worker file with `push` event handler that calls `self.registration.showNotification()`
- VAPID key pair (generate once, store public key in env, private in server-only env)
- Client: `navigator.serviceWorker.register` + `pushManager.subscribe({ userVisibleOnly: true, applicationServerKey })`, stash subscription in DB
- New DB table `push_subscriptions` (user_id, endpoint, p256dh, auth, created_at)
- Server: `web-push` npm lib, send helper keyed on user_id
- Settings UI: permission-request button + "Add to Home Screen" instruction card for iOS users (Web Push only works in installed PWA on iOS 16.4+)
- Trigger wiring at these four events:
  1. **Member check-in** — fires on successful `submitCheckInAction`, sends to all OTHER challenge members
  2. **Weekly settlement** — fires inside the cron handler after `settleWeekForChallenge` completes, sends to everyone in the challenge with their own delta
  3. **Reward redemption** — fires on successful `redeemRewardAction`, sends to all OTHER challenge members
  4. **Daily no-check-in reminder** — user sets a preferred time in Settings ("remind me at 18:00"); a new cron runs hourly and sends to users whose time bucket matches AND who haven't checked in today

Daily reminder requires a new nullable `challenge_members.reminder_hour smallint` column (single reminder time per user; morning + evening or custom multi-time deferred to v2) and a second cron entry in `vercel.json` (`0 * * * *` for hourly reminder scan — Pro plan already required for CR-02 hourly settlement cron).

Settings toggle master-switch: `notifications_enabled` boolean on `challenge_members` (or a new `user_preferences` table) gates all 4 triggers at send time. **Default value = `false`** — explicit opt-in per browser permission best practice and to avoid surprise notifications on first install.

**SETT-03 — Profile editing = Name (both DB layers) + Custom photo upload**

- **Name edit** — updates both `challenge_members.displayName` AND `auth.users.user_metadata.full_name` (Supabase admin API, server-side). Keeps identity consistent across the app.
- **Photo edit** — upload to new Supabase Storage `avatars` bucket. Client compresses to **512×512** square (retina-ready for small avatars and the larger "hero" uses in Feed header). Cache-bust via `?v=<timestamp>` query param on the stored URL so other members see updates immediately. On upload success: update `challenge_members.avatarUrl` to the new Storage URL. If no custom photo set, fall back to Google OAuth avatar URL (already captured on signup). Need new RLS policies on the bucket: public-read, users can write only their own folder (`auth.uid()::text = (storage.foldername(name))[1]`).

### Design Polish (DSGN-01, DSGN-02, DSGN-03)

**Scope = New screens only, not retroactive audit.**
DSGN requirements apply to work built in this phase (Dashboard additions, Feed build, Settings additions). The broader app polish backlog (day-dots pill redesign, navbar changes, goal stepper redesign, typography, icons, empty/error states) is explicitly deferred to `/gsd-ui-review` AFTER Phase 5 completes. Do NOT attempt a global polish pass inside this phase.

## Locked-in constraints from prior phases

- No Hype/Nudge reactions in v1 (deferred)
- Photo check-in is the only workout logging mechanic
- Stitch Royale theme: dark navy (#001233) + emerald green (#50C878), Plus Jakarta Sans, Material Symbols for nav icons, Lucide for incidental UI
- Google OAuth only (no email/password, no Apple)
- No named groups — flat connection model via invite links
- Settlement cron already runs hourly (changed in Phase 4 CR-02 fix)
- Vercel Pro plan already required for existing hourly cron

## Out of scope (defer to v2 or later)

- Hype/Nudge reactions on feed posts
- Comments/likes on check-ins
- Privacy toggles
- Multiple challenges per user
- Feed realtime subscriptions (pull-to-refresh is enough for v1)
- App-wide polish pass (tracked separately in `ui_polish_notes.md` — run `/gsd-ui-review` after Phase 5)
- Email notifications (Web Push covers the use case)
