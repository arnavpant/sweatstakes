    3. Implement `submitCheckInAction(photoUrl: string, checkedInDate: string)`:
       - Auth gate: `const supabase = await createClient(); const { data: { user } } = await supabase.auth.getUser(); if (!user) return { error: 'Not authenticated' }`
       - Validate photoUrl: must be a non-empty string. Validate with Zod: `z.string().url()`. If the env var `NEXT_PUBLIC_SUPABASE_URL` is set, also check that photoUrl starts with it. Return `{ error: 'Invalid photo URL' }` on failure.
       - Validate checkedInDate: must match YYYY-MM-DD format via regex `/^\d{4}-\d{2}-\d{2}$/`. Parse as Date and verify it's not in the future and not more than 1 day in the past. Return `{ error: 'Invalid date' }` on failure.
       - Get user's challenge: query challengeMembers where userId = user.id, limit 1. If empty, return `{ error: 'Not in a challenge' }`.
       - Insert into checkIns with userId: user.id, challengeId, photoUrl, checkedInDate.
       - Per D-17: No deduplication on insert -- unlimited check-ins per day. Deduplication happens in progress queries.
       - Per D-18: No delete action -- once submitted, permanent.
       - Return `{ success: true }`.

    4. Implement `updateWeeklyGoalAction(goal: number)`:
       - Auth gate (same pattern).
       - Validate goal with Zod: `z.number().int().min(1).max(7)`. Return `{ error: 'Goal must be between 1 and 7' }` on failure.
       - Update challengeMembers set weeklyGoal = goal where userId = user.id. Per D-09: immediate effect, no restrictions.
       - Return `{ success: true }`.

       `getWeeklyProgress(userId: string, challengeId: string, date: Date)`:
       - Compute week bounds using getWeekBounds(date).
       - Query checkIns where userId = userId AND challengeId = challengeId AND checkedInDate >= start AND checkedInDate <= end.
       - Select distinct checkedInDate values using `sql<string>\`DISTINCT ${checkIns.checkedInDate}\`` or by deduplicating in JS.
       - Query challengeMembers for weeklyGoal where userId = userId.
       - Return `{ checkedInDays: string[], goal: number }`.
       - Import `db` from `@/db` and `{ checkIns, challengeMembers }` from `@/db/schema` and `{ eq, and, gte, lte }` from `drizzle-orm`.

       `computeStreak(userId: string, challengeId: string, weeklyGoal: number)`:
       - Start from the PREVIOUS completed week (never the current week per D-16 / Pitfall 5).
       - Work backward week by week. For each week, count distinct check-in days. If count >= weeklyGoal, increment streak. If count < weeklyGoal, stop.
       - Limit backward search to 52 weeks (1 year max streak).
       - Return the streak count as a number.

    Both `getWeeklyProgress` and `computeStreak` live in `src/lib/utils/week.ts` because they are data-fetching utilities called from Server Components, not Server Actions triggered by user mutations.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client -> Server Action | Untrusted photoUrl and checkedInDate from client cross into server mutation |
| Client -> Supabase Storage | Untrusted file upload from browser to storage bucket |
| Server Action -> Database | Server action writes to checkIns and challengeMembers tables |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Spoofing | submitCheckInAction | mitigate | Auth gate via supabase.auth.getUser() -- userId from auth token, never from client input |
| T-03-02 | Tampering | submitCheckInAction photoUrl | mitigate | Validate photoUrl is a valid URL via Zod z.string().url(); optionally verify it starts with NEXT_PUBLIC_SUPABASE_URL |
| T-03-03 | Tampering | submitCheckInAction checkedInDate | mitigate | Regex validation for YYYY-MM-DD format; reject future dates and dates more than 1 day in past |
| T-03-04 | Tampering | updateWeeklyGoalAction goal | mitigate | Zod validation: z.number().int().min(1).max(7) rejects out-of-range values |
| T-03-05 | Elevation | submitCheckInAction | mitigate | Verify user has challengeMembers row before inserting check-in; prevents non-members from posting |
| T-03-06 | Denial of Service | Supabase Storage | mitigate | Client-side compression to 500KB via browser-image-compression before upload (Plan 02) |
| T-03-07 | Information Disclosure | Server Actions | mitigate | Return generic error strings, never expose SQL errors or stack traces |
</threat_model>
