**Action 1: `generateInviteLinkAction`**
- Call `createClient()` then `supabase.auth.getUser()` — return `{ error: 'Not authenticated' }` if no user
- Query `challengeMembers` for existing membership: `db.select({ challengeId: challengeMembers.challengeId }).from(challengeMembers).where(eq(challengeMembers.userId, user.id)).limit(1)`
- If membership exists, use that `challengeId`
- If no membership (D-11: implicit challenge creation):
  1. Insert into `challenges`: `const [newChallenge] = await db.insert(challenges).values({}).returning()`
  2. Insert self into `challengeMembers`: `db.insert(challengeMembers).values({ challengeId: newChallenge.id, userId: user.id, displayName: user.user_metadata.full_name || user.email || 'Member', avatarUrl: user.user_metadata.avatar_url || null })`
  3. Use `newChallenge.id` as `challengeId`
- Generate code: `const code = generateCode()`
- Calculate expiry: `const expiresAt = new Date(Date.now() + 24 * 60 * 60 * 1000)` (24 hours per D-03)
- Insert invite link: `db.insert(inviteLinks).values({ code, challengeId, createdBy: user.id, expiresAt })`
- Build URL: `const siteUrl = process.env.NEXT_PUBLIC_SITE_URL || 'http://localhost:3000'`
- Return `{ url: \`\${siteUrl}/join/\${code}\` }`

**Action 2: `joinChallengeAction(code: string)`**
- Call `createClient()` then `supabase.auth.getUser()` — return `{ error: 'Not authenticated' }` if no user
- Check existing membership (D-14): query `challengeMembers` where `userId = user.id`. If exists, return `{ error: 'already_in_challenge' }`
- **Atomic validate-and-consume** (per Pitfall 3 — prevents race condition on one-time-use codes):
  ```typescript
  const updated = await db
    .update(inviteLinks)
    .set({ usedAt: new Date() })
    .where(and(
      eq(inviteLinks.code, code),
      isNull(inviteLinks.usedAt),
      gt(inviteLinks.expiresAt, new Date())
    ))
    .returning()
  ```
- If `updated.length === 0`, return `{ error: 'invalid_or_expired' }` (D-09: covers invalid, expired, and already-used)
- Insert new member: `db.insert(challengeMembers).values({ challengeId: updated[0].challengeId, userId: user.id, displayName: user.user_metadata.full_name || user.email || 'Member', avatarUrl: user.user_metadata.avatar_url || null })`
- Return `{ success: true }`

**Action 3: `leaveChallengeAction`**
- Call `createClient()` then `supabase.auth.getUser()` — return `{ error: 'Not authenticated' }` if no user
- Delete membership: `await db.delete(challengeMembers).where(eq(challengeMembers.userId, user.id))`
- Return `{ success: true }`
- Per D-13: no special owner role, challenge continues as long as 1+ members remain. No cleanup of empty challenges needed for v1.

Every action starts with `getUser()` auth check — this is the RLS substitute since Drizzle bypasses PostgreSQL RLS (per Pitfall 1 in RESEARCH.md).

**Step 1: Create `src/app/join/[code]/page.tsx` — Public join route.**

This is a Server Component. It must be OUTSIDE the `(protected)` route group so proxy.ts does not redirect unauthenticated users. The `/join/*` path is NOT in proxy.ts's `protectedRoutes` array, so it is already public.

```typescript
import { redirect } from 'next/navigation'
import { createClient } from '@/lib/supabase/server'
import { joinChallengeAction } from '@/lib/actions/connections'

export default async function JoinPage({ params }: { params: Promise<{ code: string }> }) {
  const { code } = await params
  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()

  if (!user) {
    // D-07: Unauthenticated — redirect to login with next param preserving join intent
    redirect(`/login?next=/join/${encodeURIComponent(code)}`)
  }

  // D-08: Authenticated — attempt join immediately, no confirmation step
  const result = await joinChallengeAction(code)

  if (result.error) {
    // D-09: Show error page with reason
    redirect(`/join/${encodeURIComponent(code)}/error?reason=${encodeURIComponent(result.error)}`)
  }

  // Success — go to dashboard
  redirect('/dashboard')
}
```

IMPORTANT: In Next.js 16, `params` is a Promise that must be awaited (not a direct object). Use `const { code } = await params`.

```typescript
export async function signInWithGoogleAction(next: string = '/dashboard') {
  const supabase = await createClient()
  const requestHeaders = await headers()

  const siteUrl =
    process.env.NEXT_PUBLIC_SITE_URL ||
    requestHeaders.get('origin') ||
    'http://localhost:3000'

  // Thread the next param through OAuth so auth/callback redirects to the right place
  const { data, error } = await supabase.auth.signInWithOAuth({
    provider: 'google',
    options: {
      redirectTo: `${siteUrl}/auth/callback?next=${encodeURIComponent(next)}`,
    },
  })

  if (error || !data.url) {
    return redirect('/login?error=OAuthSigninFailed')
  }

  redirect(data.url)
}
```

The key change: `redirectTo` now includes `?next=${encodeURIComponent(next)}`. The existing `auth/callback/route.ts` already reads the `next` param and redirects there, so no changes needed to the callback route. This threads the join intent through the entire OAuth round-trip (Pitfall 4 mitigation).

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04 | Tampering | Invite code brute-force | accept | 32^8 = ~1 trillion combinations. At friend-group scale, no rate limiting needed. If needed later, add in proxy.ts. |
| T-02-05 | Tampering | Race condition on one-time codes | mitigate | Atomic UPDATE with `WHERE used_at IS NULL AND expires_at > NOW() RETURNING` — DB enforces single use. Task 1 implements this. |
| T-02-06 | Elevation of Privilege | Open redirect via `next` param | mitigate | auth/callback/route.ts already validates: `next` must start with `/` and not `//`. No changes needed — existing mitigation covers the join flow. |
| T-02-07 | Spoofing | Unauthenticated DB access | mitigate | Every Server Action calls `supabase.auth.getUser()` before any Drizzle query. Returns error if no authenticated user. Task 1 implements this on all 3 actions. |
| T-02-08 | Information Disclosure | Error message leaks | mitigate | Error page shows generic messages ("expired or already been used"), not internal details. Error codes (invalid_or_expired, already_in_challenge) are application-level, not SQL errors. Task 2 implements this. |
| T-02-09 | Elevation of Privilege | User joins challenge while already in one | mitigate | joinChallengeAction checks existing membership BEFORE consuming invite code. Returns `already_in_challenge` error. Task 1 implements this per D-14. |
